# E2 Fast Predictor — Architectural Notes

## Role in the REE Loop

E2 is the motor-sensory forward model: `f(z_t, a_t) → z_{t+1}`.

It answers the question: *"If I take action `a` from latent state `z`, what latent state will I be in next?"* This makes trajectory planning possible — HippocampalModule uses E2 to roll out candidate action sequences and score their sensory consequences.

## What E2 Operates On

**E2 does not operate on raw sensory streams.** It operates exclusively on `z_gamma` — the unified latent space of all sensory modalities, produced by the LatentStack. `z_gamma` is the level at which coherent objects form from bound multi-modal features.

This has two implications:

1. **Less processing required**: By the time input reaches E2, the sensory binding problem is already solved by the LatentStack. E2 predicts object-level transformations, not pixel/feature-level changes.

2. **Correct error signal**: E2's training target is `z_{t+1}` (the actual latent state after the action was taken and sensed), not a purely sensory prediction. This is the motor-sensory error signal — categorically distinct from E1's sensory-only prediction error.

## E2 Horizon vs E1 Horizon

E2's `rollout_horizon` must be **longer** than E1's `prediction_horizon`.

| Component | Default horizon | Predicts |
|-----------|----------------|---------|
| E1 | 20 steps | Perceived present — sensory-only, no action conditioning. E1 predicts a short way into the future to model what the agent will likely perceive. |
| E2 | 30 steps | Motor-transformed future — action-conditioned. E2 predicts how actions will transform latent sensory objects over a longer horizon to support trajectory selection. |

E1 predicts the "perceived present" by projecting the current latent stream forward a short distance. E2 needs to plan further ahead because trajectory evaluation requires simulating downstream consequences of multi-step action sequences.

## Training Signal

E2 is trained on `(z_t, action, z_{t+1})` transition tuples, where:
- `z_t` = `z_gamma` at step t (before action, detached from compute graph)
- `action` = action taken at step t
- `z_{t+1}` = `z_gamma` at step t+1 (actual sensory consequence, detached)

Loss: MSE(`E2(z_t, action)`, `z_{t+1}`).

The transition buffer in `agent.py` carries `prev_latent_z` and `prev_action_tensor` across loop iterations, recording the tuple at step t+1 once the true next latent state is available.

## Relationship to HippocampalModule

HippocampalModule owns trajectory generation. It calls E2 internally to simulate candidate trajectories. E2 is never called directly from the agent's main loop — only through HippocampalModule (SD-001 resolution).

## §4 — E1→HippocampalModule Wiring (SD-002)

E1's associative prior is passed into `HippocampalModule.propose_trajectories()` as `e1_prior`. This conditions HippocampalModule's terrain search on long-horizon associative context. E1 shapes the prior over which parts of the motor-sensory landscape to explore; E2 evaluates the actual trajectories within that landscape.

See `agent.py:generate_trajectories()` and `hippocampal/module.py:propose_trajectories()`.

## §5 — V3 Target: Action Objects and Hippocampal Map Compression (SD-004)

> **Status: documented design target for V3. Not yet implemented.**

### The core insight

E2 currently predicts `z_{t+1}` directly. In V3, E2 should also produce **action objects** — compressed latent representations of the transformation `(z_t → z_{t+1})` under action `a_t`. These action objects become the primitive vocabulary of the hippocampal map.

Instead of: `f(z_t, a_t) → z_{t+1}`

V3 target: `f(z_t, a_t) → (z_{t+1}, o_t)` where `o_t` is an action object.

An action object `o_t` captures *what this action does* in latent space — not just the outcome state, but the abstract structure of the transformation itself. It is a compressed bottleneck inside E2's transition model.

### What action objects enable

**1. Hippocampal map over action objects, not raw states**

The hippocampal map currently navigates `z_gamma` state space. State space is high-dimensional and the map grows with every visited state. With action objects, the hippocampal map becomes a graph over action-object space:
- Nodes are action objects (what transformations are available)
- Edges connect action objects that can be sequenced
- Similar actions from similar states share the same action object representation

This is a graph of affordances, not a graph of states. It is fundamentally more compact because many distinct states can afford the same abstract transformation.

**2. Compaction of E2 hidden layer complexity**

E2's hidden layers do not need to maintain full `z_gamma` dimensionality throughout. The action object is a compressed bottleneck: `(z_t, a_t) → o_t → z_{t+1}`. Because `o_t` is lower-dimensional, E2 can be smaller while still supporting the reconstruction `o_t → z_{t+1}`. The transition model encodes "what kind of action this is" in `o_t`, then expands it back to the state space for the next-state prediction.

**3. Much longer hippocampal rollout horizons**

CEM rollouts currently operate step-by-step in `z_gamma` space. In V3, HippocampalModule navigates in action-object space. Each step in the map is an action object, not a raw state. This means:
- The planning horizon can extend much further without proportional cost
- Long multi-step plans become sequences of action objects, unfolded to actual actions at execution time
- The hippocampal map becomes the long-horizon plan; E2 rollouts supply the short-horizon local transitions for scoring

### Sketch of required changes

**E2FastPredictor:**
- Add `action_object_dim` to `E2Config` (likely smaller than `latent_dim`)
- Refactor `transition` into two sub-networks:
  - `action_encoder`: `(z_t, a_t) → o_t` — the bottleneck/action object
  - `transition_decoder`: `(z_t, o_t) → z_{t+1}` — state reconstruction from action object
- `predict_next_state` passes through both; `o_t` is exposed as a return value
- New method: `encode_action_object(z_t, a_t) → o_t`
- New method: `decode_action_object(z_t, o_t) → z_{t+1}`
- Training loss: MSE on `z_{t+1}` reconstruction (same as now, but through bottleneck)

**HippocampalModule:**
- Replace state-space terrain prior with action-object-space prior
- Map nodes stored in action-object space (`o_t` vectors), not `z_gamma`
- CEM operates by proposing action-object sequences, not raw action sequences
- Unfold action-object plans to actual actions at execution time via `decode_action_object`
- Terrain scoring operates on action-object sequences (lower-dimensional → longer horizon possible)

**Trajectory:**
- Add `action_objects: Optional[List[torch.Tensor]]` field to carry `o_t` sequence alongside `states` and `actions`

**HippocampalConfig:**
- Add `action_object_dim: int` to match `E2Config.action_object_dim`
- `horizon` can be increased substantially once map operates in object space

### Relationship to existing neuroscience models

This pattern is analogous to:
- **Place cells / grid cells**: action objects as place-coded affordance representations
- **Options framework**: action objects ≈ option abstractions with compressed outcome models
- **Bisimulation metrics**: states with identical action-object mappings are behaviourally equivalent and can be merged in the hippocampal map

The hippocampal module comment in `module.py` (V3 considerations) already anticipated this: "a more principled design would model the hippocampus as learning a map of the affective landscape and navigating it via place-cell-like representations." Action objects are the place-cell-like representations — indexed by action affordance rather than raw spatial location.
